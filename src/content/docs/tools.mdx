Browy ships with the **browser-driving tools enabled by default**.
You can turn off any individual tool in **Settings → Tools** — handy
for running in stricter modes (read-only, no JS-eval, etc.).
Host-touching tools (`read_file`, `bash`, `web_fetch`) are **only
available** through the headless terminal CLI ([`browy run`](#host-tools-terminal-cli)),
never inside the browser surfaces.

<Aside type="note">
The authoritative list lives in
[`src/agent/tools/browser.ts`](https://github.com/BrowyHQ/browy/blob/main/src/agent/tools/browser.ts) —
search for `register({` to see every tool.
</Aside>

## Categories

<Tabs>
<TabItem label="Vision">

| Tool | What it does |
|---|---|
| `snapshot` | Take an indexed accessibility-tree snapshot of the active tab. The agent uses these `[N]` indexes to click and type without brittle CSS. |
| `screenshot` | Capture a PNG of the visible viewport (or full page). |
| `read_visible` | Return the visible text of the page. |

</TabItem>
<TabItem label="Click / type / nav">

| Tool | What it does |
|---|---|
| `click(index)` | Click an element by its snapshot index. |
| `type(index, text)` | Focus an input by index and type into it. |
| `key(key)` | Press a keyboard key (`Enter`, `Escape`, `Tab`, …). |
| `scroll(direction, px?)` | Scroll the page or a scroll container. |
| `navigate(url)` | Open a URL (in a new tab by default). |

</TabItem>
<TabItem label="Tabs">

| Tool | What it does |
|---|---|
| `list_tabs` | List all open tabs (title, URL, id). |
| `switch_tab(id)` | Bring a tab to the foreground. |
| `new_tab(url?)` | Open a new tab. |
| `close_tab(id)` | Close a tab. |

</TabItem>
<TabItem label="Power tools">

| Tool | What it does |
|---|---|
| `evaluate_js(code)` | Run arbitrary JavaScript in the page (top-level `await` OK). For shadow-DOM walks, custom waits, structured extraction, framework-specific APIs. |
| `run_script(code)` | Run a Node.js script on **your** machine for file I/O / shell. Sandboxed and asks before running. |
| `query_dom(selector)` | CSS selector fallback when the indexed tools can't express what you need. |

</TabItem>
<TabItem label="Network & console">

| Tool | What it does |
|---|---|
| `network_log` | Recent network requests (URL, method, status, timing, headers). |
| `console_log` | Recent console messages and exceptions. |

</TabItem>
<TabItem label="Disk + memory">

Sandboxed at `~/.browy/data/`:

| Tool | What it does |
|---|---|
| `save_file(filename, content)`, `read_file`, `list_files`, `delete_file` | Scratch disk for cached scrapes, intermediate work. Paths are relative to the data root; you can't escape it. |
| `note_set(key, value, category?)`, `note_get`, `note_list`, `note_delete` | Persistent key-value memory that survives across **different chats** (user preferences, long-lived facts). |

<Aside type="caution" title="Notes vs conversation history">
`note_*` is **not** the agent's conversation history — that's automatic
and the agent always has it. Empty `note_list` does not mean empty
context.
</Aside>

</TabItem>
<TabItem label="Human-in-loop">

| Tool | What it does |
|---|---|
| `await_user(message)` | Pause and ask you to do something yourself (solve a CAPTCHA, click a confirmation, choose between two options). |

</TabItem>
</Tabs>

## Configurable per-tool toggles

Every built-in browser tool can be turned off individually in
**Settings → Tools** (right-click the Browy toolbar icon → Options).
Toggles are grouped by category — _inspect_, _navigate_, _interact_,
_devtools_ — and persist in `chrome.storage.local`. Disabled tools are
stripped from the SDK's allowlist **and** from the function-definition
list sent to the model, so the agent never sees they exist.

Common reasons to disable a tool:

<CardGrid>
  <Card title="Stricter sandbox" icon="seti:lock">
    Turn off `evaluate_js` and `run_script` if you don't want the agent
    running arbitrary JavaScript inside the inspected page (it'll still
    use the structured tools — `click_index`, `fill_form`, etc.).
  </Card>
  <Card title="Read-only mode" icon="seti:eye">
    Disable everything in _interact_ and _navigate_; keep _inspect_ on.
    The agent can answer questions about pages but can't change them.
  </Card>
  <Card title="No file I/O" icon="seti:folder">
    Disable `download_file` / `upload_file` if you never want the agent
    triggering picker dialogs.
  </Card>
</CardGrid>

Click **enable all** in the toolbar to undo every override at once.
Changes apply on the next chat (or after `/clear`).

## Host tools (terminal CLI)

The Copilot SDK ships `read_file` / `write_file` / `bash` / `grep` /
`glob` / `web_fetch` for true coding-agent workflows. These touch your
**host machine**, so Browy's browser surfaces (side panel, DevTools
panel) **never** expose them — only the browser tools above are
available there.

To use the host toolset, run the headless terminal CLI:

```bash
browy run "summarize the diff staged in this repo"
browy run "find every TODO in src/ and group them by file"
browy run                                  # interactive REPL
browy run --resume <session-id>            # pick up where you left off
```

`browy run` is scoped to your current directory by default
(`--cwd <path>` to override). Sessions are stored under
`~/.browy/cli-sessions/`, isolated from both Browy's browser sessions
and your own `copilot` CLI sessions. See
[**DevTools CLI →**](/docs/devtools-cli/) for the full surface.
