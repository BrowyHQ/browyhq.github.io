Browy ships with the browser-driving tools enabled by default. Any
individual tool can be turned off in **Settings → Tools** for stricter
modes (read-only, no JS evaluation, and so on). Browy's browser
surfaces (side panel + DevTools panel) only ever expose browser tools;
the Copilot SDK's host-touching tools (`bash`, `read_file`,
`write_file`, `web_fetch`) are blocked by a strict allowlist and never
reach the model — even if a malicious page tries to call them.

<Aside type="note">
The authoritative list lives in
[`src/agent/tools/browser.ts`](https://github.com/BrowyHQ/browy/blob/main/src/agent/tools/browser.ts).
Search for `register({` to see every tool.
</Aside>

## Categories

<Tabs>
<TabItem label="Vision">

| Tool | What it does |
|---|---|
| `snapshot` | Indexed accessibility-tree snapshot of the active tab. The agent uses the `[N]` indexes to click and type without CSS selectors. |
| `screenshot` | PNG of the visible viewport (or full page). |
| `read_visible` | Visible text of the page. |

</TabItem>
<TabItem label="Click / type / nav">

| Tool | What it does |
|---|---|
| `click(index)` | Click an element by its snapshot index. |
| `type(index, text)` | Focus an input by index and type into it. |
| `key(key)` | Press a keyboard key (`Enter`, `Escape`, `Tab`, …). |
| `scroll(direction, px?)` | Scroll the page or a scroll container. |
| `navigate(url)` | Open a URL (in a new tab by default). |

</TabItem>
<TabItem label="Tabs">

| Tool | What it does |
|---|---|
| `list_tabs` | List open tabs (title, URL, id). |
| `switch_tab(id)` | Bring a tab to the foreground. |
| `new_tab(url?)` | Open a new tab. |
| `close_tab(id)` | Close a tab. |

</TabItem>
<TabItem label="Page scripting">

| Tool | What it does |
|---|---|
| `evaluate_js(code)` | Run JavaScript in the page (top-level `await` works). For shadow-DOM walks, custom waits, structured extraction. |
| `query_dom(selector)` | CSS selector fallback when the indexed tools can't reach what you need. |

</TabItem>
<TabItem label="Network & console">

| Tool | What it does |
|---|---|
| `network_log` | Recent network requests (URL, method, status, timing, headers). |
| `console_log` | Recent console messages and exceptions. |

</TabItem>
<TabItem label="Disk + memory">

Sandboxed at `~/.browy/data/`:

| Tool | What it does |
|---|---|
| `save_file(filename, content)`, `read_file`, `list_files`, `delete_file` | Scratch disk for cached scrapes and intermediate work. Paths are relative to the data root and can't escape it. |
| `note_set(key, value, category?)`, `note_get`, `note_list`, `note_delete` | Persistent key-value memory that survives across different chats (user preferences, long-lived facts). |

<Aside type="caution" title="Notes vs conversation history">
`note_*` is not the agent's conversation history. The agent always has
its own conversation history; an empty `note_list` does not mean empty
context.
</Aside>

</TabItem>
<TabItem label="Human-in-loop">

| Tool | What it does |
|---|---|
| `await_user(message)` | Pause and ask you to do something yourself (solve a CAPTCHA, click a confirmation, choose between two options). |

</TabItem>

## Configurable per-tool toggles

Every browser tool can be turned off individually in **Settings →
Tools** (right-click the Browy toolbar icon → Options). Toggles are
grouped by category — _inspect_, _navigate_, _interact_, _devtools_ —
and persist in `chrome.storage.local`. Disabled tools are stripped from
the SDK's allowlist and from the function-definition list sent to the
model, so the agent doesn't see they exist.

Common reasons to disable a tool:

<CardGrid>
  <Card title="Stricter sandbox" icon="seti:lock">
    Turn off `evaluate_js` if you don't want the agent running
    JavaScript in the inspected page. The structured tools (`click`,
    `type`, `navigate`, etc.) keep working.
  </Card>
  <Card title="Read-only mode" icon="seti:eye">
    Disable everything in _interact_ and _navigate_ and keep _inspect_
    on. The agent can answer questions about pages but can't change
    them.
  </Card>
  <Card title="No JS evaluation" icon="seti:javascript">
    Some teams disable `evaluate_js` and `query_dom` to limit the agent
    to the structured accessibility-tree tools only.
  </Card>
</CardGrid>

Click **enable all** in the toolbar to undo every override at once.
Changes apply on the next chat (or after `/clear`).

## What about host shell, filesystem, web_fetch?

The Copilot SDK includes `bash`, `read_file`, `write_file`,
`web_fetch`, `grep`, and `glob` as standard tools — they're how the
`copilot` CLI does coding work in a terminal. Browy's browser surfaces
do **not** expose any of them, by design. The native host registers a
strict allowlist when it opens a Copilot session, so even if a model
tried to call `bash` in response to a prompt-injected page, the SDK
would reject it. There is no Settings toggle to turn them on, and no
in-browser path to opt in.

If you need a terminal coding agent, use the
[GitHub Copilot CLI](https://docs.github.com/en/copilot/github-copilot-in-the-cli)
directly.
